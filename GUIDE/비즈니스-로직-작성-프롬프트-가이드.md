## 📝 작성 규칙

### LogicID 생성 규칙
1. **형식**: `BL-{도메인}-{순번}`
2. **도메인 매핑**:
   - CALENDAR → CALENDAR
   - ATTENDANCE → ATTENDANCE
   - MEMBER → MEMBER
   - AGENCY → AGENCY
   - MANAGER → MANAGER
   - HEALTH → HEALTH
   - PROJECT → PROJECT
3. **순번**: 해당 도메인 내에서 001부터 시작하여 순차적으로 증가
4. **예시**:
   - `BL-ATTENDANCE-001`: 휴가 신청 승인 처리
   - `BL-CALENDAR-001`: 일정 추가
   - `BL-MEMBER-001`: 회원가입

### 도메인 분류 규칙
1. **CALENDAR**: 캘린더 관련 로직 (일정 추가/수정/삭제, 메모 등)
2. **ATTENDANCE**: 근태 관련 로직 (출근/퇴근, 휴가 신청/승인, 워케이션 등)
3. **MEMBER**: 회원 관련 로직 (회원가입, 로그인, 프로필 수정 등)
4. **AGENCY**: 에이전시 관련 로직 (에이전시 관리, 가입 요청 승인 등)
5. **MANAGER**: 담당자 관련 로직 (담당자 배정, 프로젝트 관리 등)
6. **HEALTH**: 건강 관리 관련 로직 (건강 설문, 검진 등)
7. **PROJECT**: 프로젝트 관련 로직 (프로젝트 생성/수정, 칸반 보드, 작업 관리 등)

### 트랜잭션 유형 판단 규칙
1. **READ_ONLY**: 
   - 데이터 조회만 수행
   - 통계 계산 (기존 데이터 기반)
   - 필터링/정렬
   - 예시: 대시보드 통계 조회, 목록 조회
2. **READ_WRITE**:
   - 데이터 생성 (Insert)
   - 데이터 수정 (Update)
   - 데이터 삭제 (Delete)
   - 상태 변경
   - 예시: 회원가입, 휴가 승인, 프로젝트 생성

### 영향받는 테이블 작성 규칙
1. **작업 유형 표시**: [Insert], [Update], [Delete], [Select]로 명확히 표시
2. **테이블명**: ERD의 실제 테이블명 사용 (대문자, 언더스코어)
3. **설명**: 각 테이블에 어떤 작업이 수행되는지 간단히 설명
4. **순서**: Insert → Update → Delete → Select 순서로 정렬

### 진행상태 설정 규칙
1. **개발전**: 아직 설계도 시작하지 않은 상태
2. **설계중**: 현재 문서 작성 중인 상태 (기본값)
3. **개발중**: 개발이 진행 중인 상태
4. **코드리뷰**: 개발 완료 후 코드 리뷰 중인 상태
5. **완료**: 개발 및 리뷰 완료, 배포 가능한 상태

## 🔍 작성 예시

### 예시 1: 휴가 신청 승인 처리

```
## LogicID: BL-ATTENDANCE-001

## 도메인
ATTENDANCE

## 요구사항 코드
AG-AC-12

## 진행상태
설계중

## 트랜잭션 유형
READ_WRITE

## 영향받는 테이블[엔티티]
- [Update] `ATTENDANCE_REQUEST`: 신청 상태를 'PENDING'에서 'APPROVED'로 변경
- [Update] `LEAVE_BALANCE`: 연차 잔여일 차감 (USED_DAYS 증가, REMAIN_DAYS 감소)
- [Insert] `CALENDAR_EVENT`: 승인된 휴가 일정을 캘린더에 등록
- [Insert] `NOTIFICATION`: 신청자에게 승인 알림 발송

## 1. 개요 (Context)
- **목적**: 사용자가 휴가/워케이션을 신청했을 때, 관리자가 이를 승인하면 일어나는 잔여 연차 차감 및 캘린더 반영 로직
- **Trigger**: 관리자가 승인 버튼을 클릭했을 때

## 2. 데이터 흐름 (I/O)
- **Input Data (DTO)**: 
  - `requestNo`: NUMBER - 신청 번호
  - `approverNo`: NUMBER - 승인자 번호
  - `status`: VARCHAR2(20) - 'APPROVED'
- **Output Data**: 
  - `Boolean` - 성공 여부 또는 `UpdatedEntityDTO`

## 3. 상세 프로세스 (Business Logic)

### 3.1 유효성 검증 (Validation)
1. 신청 건 존재 확인
   - 조건: `ATTENDANCE_REQUEST` 테이블에서 `requestNo`로 조회
   - 실패 시: `ResourceNotFoundException` 발생
2. 신청 상태 확인
   - 조건: `ATTENDANCE_REQUEST_STATUS = 'PENDING'`
   - 실패 시: `AlreadyProcessedException` 발생
3. 잔여 연차 확인 (휴가일 경우)
   - 조건: `LEAVE_BALANCE_REMAIN_DAYS >= ATTENDANCE_REQUEST_USING_DAYS`
   - 실패 시: `BalanceInsufficientException` 발생

### 3.2 상태 변경 (State Transition)
- 이전 상태: `ATTENDANCE_REQUEST_STATUS = 'PENDING'`
- 변경 후 상태: `ATTENDANCE_REQUEST_STATUS = 'APPROVED'`

### 3.3 데이터 처리 (Data Processing)
1. 신청 상태 변경
   - 작업: `ATTENDANCE_REQUEST` 테이블의 상태를 'APPROVED'로 변경
   - 테이블: `ATTENDANCE_REQUEST`
2. 연차 차감
   - 작업: `LEAVE_BALANCE` 테이블에서 `USED_DAYS` 증가, `REMAIN_DAYS` 감소
   - 테이블: `LEAVE_BALANCE`
3. 캘린더 반영
   - 작업: `CALENDAR_EVENT` 테이블에 해당 일정 자동 등록
   - 테이블: `CALENDAR_EVENT`
4. 알림 생성
   - 작업: 신청자에게 승인 알림 발송
   - 테이블: `NOTIFICATION`

### 3.4 계산 로직 (Calculation)
- 잔여 연차 계산:
  ```
  REMAIN_DAYS = TOTAL_DAYS - USED_DAYS
  ```
- 사용 일수 계산:
  ```
  USING_DAYS = END_DATE - START_DATE + 1
  ```

### 3.5 연관 로직 (Related Logic)
- **이후 실행 로직**: 
  - 캘린더 일정 자동 등록
  - 신청자에게 승인 알림 발송
  - 담당자에게 알림 발송 (해당 작가의 담당자인 경우)

## 4. 영향받는 테이블 (Affected Tables)
- **[Update]** `ATTENDANCE_REQUEST`: 신청 상태 변경
- **[Update]** `LEAVE_BALANCE`: 연차 차감
- **[Insert]** `CALENDAR_EVENT`: 일정 등록
- **[Insert]** `NOTIFICATION`: 신청자에게 승인 알림 발송

## 5. 예외 처리 (Exception Handling)
- `BalanceInsufficientException`: 잔여 연차가 부족한 경우
  - 발생 조건: `REMAIN_DAYS < USING_DAYS`
  - 처리 방법: 승인 거부, 사용자에게 알림
  - 에러 메시지: "잔여 연차가 부족합니다. (잔여: N일, 신청: M일)"
- `AlreadyProcessedException`: 이미 승인/반려된 건인 경우
  - 발생 조건: `STATUS != 'PENDING'`
  - 처리 방법: 승인 거부
  - 에러 메시지: "이미 처리된 신청입니다."
- `ResourceNotFoundException`: 신청 건이 존재하지 않는 경우
  - 발생 조건: `requestNo`로 조회 시 결과 없음
  - 처리 방법: 에러 반환
  - 에러 메시지: "신청 내역을 찾을 수 없습니다."

## 6. SQL/Code Logic (Pseudo)
> JPA Dirty Checking 혹은 QueryDSL 사용 예상 구문 작성

```java
// 1. 신청 건 조회 및 검증
AttendanceRequest request = attendanceRequestRepository.findById(requestNo)
    .orElseThrow(() -> new ResourceNotFoundException("신청 내역을 찾을 수 없습니다."));

if (!request.getStatus().equals(AttendanceRequestStatus.PENDING)) {
    throw new AlreadyProcessedException("이미 처리된 신청입니다.");
}

// 2. 잔여 연차 확인 (휴가일 경우)
if (request.getType().equals(AttendanceRequestType.ANNUAL)) {
    LeaveBalance balance = leaveBalanceRepository
        .findByMemberNoAndYear(request.getMemberNo(), LocalDate.now().getYear())
        .orElseThrow(() -> new ResourceNotFoundException("연차 정보를 찾을 수 없습니다."));
    
    if (balance.getRemainDays() < request.getUsingDays()) {
        throw new BalanceInsufficientException(
            "잔여 연차가 부족합니다. (잔여: " + balance.getRemainDays() + "일, 신청: " + request.getUsingDays() + "일)"
        );
    }
    
    // 3. 연차 차감
    balance.useDays(request.getUsingDays());
    leaveBalanceRepository.save(balance);
}

// 4. 신청 상태 변경
request.approve(approverNo);
attendanceRequestRepository.save(request);

// 5. 캘린더 일정 등록
CalendarEvent event = CalendarEvent.builder()
    .memberNo(request.getMemberNo())
    .eventName(request.getType().getDisplayName())
    .startDate(request.getStartDate())
    .endDate(request.getEndDate())
    .eventType(CalendarEventType.ATTENDANCE)
    .build();
calendarEventRepository.save(event);

// 6. 알림 생성
Notification notification = Notification.builder()
    .memberNo(request.getMemberNo())
    .notificationName("휴가 승인")
    .notificationText("휴가 신청이 승인되었습니다. (기간: " + request.getStartDate() + " ~ " + request.getEndDate() + ")")
    .notificationType(NotificationType.APPROVAL)
    .build();
notificationRepository.save(notification);
```

- **이후 실행 로직**: 
  - 프로젝트 완료율 자동 계산
  - 담당자에게 완료 알림 (선택적)

## 4. 영향받는 테이블 (Affected Tables)
- **[Update]** `KANBAN_CARD`: 상태값 및 수정시간 변경
- **[Insert]** `TASK_HISTORY`: 변경 이력 추가

## 5. 예외 처리 (Exception Handling)
- `ResourceNotFoundException`: 잘못된 `taskNo`로 요청했을 때
  - 발생 조건: `taskNo`로 조회 시 결과 없음
  - 처리 방법: 에러 반환
  - 에러 메시지: "카드를 찾을 수 없습니다."

## 6. SQL/Code Logic (Pseudo)
> JPA Dirty Checking 혹은 QueryDSL 사용 예상 구문 작성

```java
// 1. 조회
Task task = taskRepository.findById(taskNo)
    .orElseThrow(() -> new ResourceNotFoundException("카드를 찾을 수 없습니다."));

// 2. 변경 (Dirty Checking에 의해 자동 Update)
task.changeStatus(TaskStatus.DONE); 

// 3. 이력 저장
TaskHistory history = TaskHistory.builder()
    .taskNo(taskNo)
    .beforeTaskHistory(task.getStatus().name())
    .afterTaskHistory(TaskStatus.DONE.name())
    .memberNo(memberNo)
    .build();
historyRepository.save(history);

## 🔗 프론트엔드 API 구조 연동

프론트엔드 코드는 `src/api/services.js`에 도메인별 서비스 객체로 구성되어 있으며, 비즈니스 로직의 도메인 분류와 일치합니다:

- **도메인 매핑**: 비즈니스 로직의 도메인과 프론트엔드 API 서비스가 1:1 매핑
  - `MANAGER` → `managerService` (담당자 관련 API)
  - `ATTENDANCE` → `attendanceService` (근태 관련 API)
  - `MEMBER` → `memberService` (회원 관련 API)
  - `AGENCY` → `agencyService` (에이전시 관련 API)
  - `HEALTH` → `healthService` (건강 관리 관련 API)
  - `PROJECT` → `projectService` (프로젝트 관련 API)
  - `LEAVE` → `leaveService` (연차/휴가 관련 API, ATTENDANCE 도메인의 하위 개념)

- **API 엔드포인트**: `src/api/config.js`의 `API_ENDPOINTS` 객체에 정의
- **사용 방법**: `import { managerService } from '@/api';` 형태로 import하여 사용

**참고**: 프론트엔드 API 구조는 `REFACTORING_GUIDE.md`의 "6.9 API 구조 및 네이밍 컨벤션" 섹션 참고
